# Архитектура VPN через Telegram-бота

## Цель

Сделать простую и устойчивую схему, где клиент оплачивает доступ в Telegram, получает WireGuard-конфиг и дальше может продлевать или отключать услугу без ручной работы.

## Базовая схема

Компоненты:

1. `Telegram Bot`
2. `Bot API / backend`
3. `PostgreSQL`
4. `WireGuard node`
5. `Payment provider`

## Как это должно работать

1. Пользователь пишет боту.
2. Бот создает заказ.
3. Пользователь оплачивает подписку.
4. Backend получает подтверждение оплаты.
5. Backend создает клиента WireGuard на нужном сервере.
6. Backend сохраняет выданный IP, ключи, срок подписки и сервер.
7. Бот отправляет клиенту `.conf` и QR-код.
8. По окончании подписки доступ либо продлевается, либо клиент отключается.

## Минимально стабильный стек

Чтобы не переусложнять, я бы рекомендовал такой набор:

1. `Ubuntu 22.04`
2. `WireGuard`
3. `Python 3.11`
4. `FastAPI`
5. `aiogram 3`
6. `PostgreSQL`
7. `systemd`
8. `Nginx`

## Почему именно так

`FastAPI` удобно принимает webhook от Telegram и webhook от платежки.

`PostgreSQL` надежнее `SQLite`, если у тебя будут оплаты, статусы подписок и несколько серверов.

`systemd` и `Nginx` дают простой production-runtime без лишней оркестрации.

## Что лучше не делать на старте

Не начинай с Kubernetes, Docker Swarm, микросервисов и отдельной очереди задач.

Не храни WireGuard-состояние только в памяти бота, иначе после рестарта будут ошибки в выдаче клиентов.

Не генерируй конфиги вручную внутри Telegram-логики без записи в БД.

## Простая production-схема

### Вариант 1. Один сервер

Подходит для MVP и первых клиентов.

1. Один VPS
2. На нем:
   bot backend
   PostgreSQL
   WireGuard
   Nginx

Плюс:

Минимум сложности и дешево.

Риск:

Один сервер = одна точка отказа.

### Вариант 2. Рекомендуемый старт

Лучший баланс простоты и надежности.

1. `Server A`:
   Telegram backend + PostgreSQL + Nginx
2. `Server B`:
   WireGuard RU/primary

Плюс:

При падении VPN-ноды не теряется бот и платежная логика.

Риск:

Нужно аккуратно настроить SSH/API доступ backend -> VPN node.

## Как backend должен управлять WireGuard

Самый простой и надежный путь на старте:

1. backend вызывает shell-скрипт на VPN-сервере
2. shell-скрипт создает клиента
3. shell-скрипт возвращает путь к конфигу, IP и имя клиента

Не надо сразу писать сложный internal API между backend и WireGuard.

## Два варианта интеграции

### Вариант A. SSH-вызов скриптов

Backend подключается к VPN-серверу по SSH и запускает:

1. `wg-add-client`
2. `wg-disable-client`
3. `wg-remove-client`

Плюс:

Быстро, понятно, стабильно для MVP.

Минус:

Нужно правильно выдать SSH-ключ и ограничить sudo.

### Вариант B. Локальный internal API на VPN-node

На VPN-сервере поднимается маленький API-сервис, который выполняет операции с WireGuard.

Плюс:

Проще масштабировать на несколько нод.

Минус:

Чуть больше кода и больше поверхности атаки.

## Что рекомендую тебе

Стартовать с `SSH + bash scripts + PostgreSQL`.

Это самый короткий путь до рабочего сервиса с оплатой и выдачей ключей.

## Жизненный цикл подписки

Статусы лучше держать простыми:

1. `pending`
2. `paid`
3. `active`
4. `expired`
5. `revoked`
6. `canceled`

## Как выдавать клиентов безопасно

Для имени клиента используй не username Telegram, а внутренний идентификатор вроде:

`u12345-s1`

Так меньше проблем с пробелами, кириллицей и конфликтами имен.

## Что важно для безопасности

1. Бот не должен иметь root-доступ ко всему серверу.
2. Для SSH-вызова лучше ограничить sudo только на конкретные скрипты.
3. Приватные ключи клиентов лучше хранить краткоживуще или в зашифрованном виде.
4. Не отправляй один и тот же конфиг повторно без причины.
5. Обязательно логируй: кто, когда, на каком сервере, какой IP получил.

## Что может сломаться

Если webhook платежки придет повторно, можно случайно создать двух клиентов.

Поэтому операции активации подписки должны быть идемпотентными.

Если бот упадет между оплатой и выдачей, статус в БД должен позволить безопасно повторить выдачу.

Если VPN-нода недоступна, заказ должен остаться в статусе `paid`, но не потеряться.

## Балансировка между несколькими VPN-серверами

На старте хватит простого правила:

1. выбирать первый активный сервер
2. либо выбирать сервер с наименьшим числом активных клиентов

Не усложняй это до реального load balancer на первом этапе.

## Следующий практический шаг

Я бы делал в таком порядке:

1. привести в порядок WireGuard-скрипты под вызов из backend
2. добавить скрипт отключения и удаления клиента
3. описать контракт ответа скриптов в JSON
4. после этого уже писать Telegram backend
